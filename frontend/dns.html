The system it goes beyond

DNS is the model — and the foil.

The paper is titled Beyond DNS for a reason: DNS is the most successful naming system ever built, and NANDA borrows almost all of its operational wisdom. But DNS was designed in 1983 for static, human-operated hosts. That assumption breaks for AI agents that move every few seconds, need to prove what they can do (not just who owns them), and demand sub-second revocation. This page covers how DNS works, what keeps it reliable (which NANDA keeps), where it falls short for agents, and whether NANDA can be affordable as an index.

Background

How DNS works in 30 seconds

  • Maps a name → a static address (e.g. example.com → an IP (Internet Protocol) address via an A/AAAA address record), plus MX (Mail eXchange), TXT (text), and NS (Name Server) records.
  • Hierarchical delegation: root servers (.) → TLD (Top-Level Domain) servers (.com) → the domain's authoritative name server. No single authority owns the whole tree.
  • Resolution: your device asks a lookup server — a recursive resolver run by your ISP (Internet Service Provider), or 8.8.8.8 / 1.1.1.1 — and that server walks root → TLD → the domain's own server, then caches the answer.
  • Caching governed by TTLs (Time To Live — how long an answer can be reused before it must be looked up again) at every layer — most queries never reach an authoritative server. This is the secret to DNS's scale.
  • Scale today: ~300M+ domains, ~10B+ lookups/day — but they're mostly static records updated minutes-to-hours apart.

DNS isn't infallible

Real failures that drove its defences

  • Cache poisoning (Kaminsky, 2008) — attackers could inject forged records into resolver caches. Drove DNSSEC (record signing). NANDA's lesson: sign everything by default, so a cache can never serve a forged answer.
  • The Dyn DDoS (2016) — a botnet flooded a major DNS provider and took down Twitter, Netflix, Reddit, etc. A reminder that even a distributed system has hot spots. NANDA's lesson: aggressive edge caching + anycast + signed-cacheable records so a downed origin doesn't stop resolution.
  • Slow propagation — a TTL of hours means a changed record can be stale globally for hours. Fine for a website; fatal for an agent that moved 5 seconds ago. NANDA's lesson: short, tiered TTLs + push volatile state off the index.

The pattern: each DNS weakness was patched after the fact (DNSSEC, DoH, anycast hardening). NANDA bakes those lessons in from the start — signing, privacy paths, and dynamism are core, not bolt-ons.

Threat model

How DNS is attacked — and whether the same tactic works on NANDA

The decisive difference is that NANDA signs everything and verifies client-side. Because every record is signed, forging or altering one gets caught and rejected — so the entire class of integrity/authenticity attacks that plague DNS stops working. What signing can't fix: keeping the service reachable (availability), guarding the secret signing keys (key management), and stopping look-alike names. Those are managed, not magically solved.

AttackEffect on DNSNANDAWhy / defence
Cache poisoning / spoofing (Kaminsky) Inject forged records into a resolver cache → users sent to a malicious IP. ✗ blocked A poisoned entry fails Ed25519/VC verification → rejected. DNS needed optional DNSSEC; NANDA signs by default.
On-path MITM tampering Alter a response in transit. ✗ blocked Any mutation breaks the signature → client refuses (this is the tamper-detection demo).
BGP (Border Gateway Protocol) / route hijack to a rogue resolver Reroute lookups to an attacker's server. ✗ blocked (integrity) The rogue can't produce valid signatures, so forged data is rejected. (It can still deny service — that's availability, below.)
DDoS / amplification Flood the infrastructure (Dyn 2016); abuse open resolvers to amplify. △ partial Still possible against any service. Edge caches, anycast, and a CDN absorb the flood. Signed, cacheable records keep serving even while the origin is under attack. Rate-limiting stops your servers from being used as an amplifier — a relay an attacker abuses to multiply traffic against a victim.
Replay (serve an old valid record) Re-present a stale-but-signed answer. △ bounded TTL + fresh issued_at + short-lived VCs (<5 min) cap the window. The paper names "replay window risk" as a known residual.
Signing-key compromise Steal a zone key → forge anything in that zone. ⚠ serious Same risk if the index/agent/resolver key leaks. Defend with KMS/HSM, rotation, fast revocation, and 3-key blast-radius isolation.
Typosquatting / look-alike names Register gooogle.com to fool users. △ residual A look-alike agent name is still registerable, but it won't carry the right issuer / credentials / reputation — trust policy (only accepting approved issuers, plus a trust-and-reputation score) catches it. A discovery-layer problem, mitigated not eliminated.
Dangling pointer / endpoint takeover A record points to a reclaimed host an attacker now controls. △ residual The signature is still valid because the record was signed honestly — but an attacker has since taken over the server it points to, so it now routes you to a malicious host. Signing can't catch this. Mitigate with short TTLs, revocation, and agent re-publish/rotation.
Enumeration / zone-walking Scrape all records (DNSSEC NSEC (Next Secure record) walking). △ open question The index can be intentionally opaque to prevent agent enumeration — but that trades against discoverability (paper §XI.B, unresolved).
Surveillance of lookups Observers learn who asks for what (DoH/DoT only partial). ✗ designed out The privacy path (PrivateFactsURL via Tor/IPFS) hides the requester by design — something DNS only bolted on late.

The pattern: NANDA converts DNS's worst attacks — poisoning, spoofing, MITM, route-hijack — from "silent compromise" into "detected and rejected," because forged bytes can't satisfy a signature. What it doesn't magically fix is availability (DDoS), key custody, replay (bounded by TTL), and naming/endpoint trust — those are managed with caching, HSMs, short TTLs, revocation, and trust policy. A good answer in the room names both halves: what the crypto eliminates, and what still needs operational care.

NANDA's own attack surface

New risks DNS doesn't even have

Signing kills the forgery class above — poisoning, MITM, tampering all fail verification. But because NANDA adds an identity, issuer, registration and capability layer on top of resolution, it inherits attack vectors DNS never had. These are the ones signing can't catch — they need key custody, federation, registration governance, and trusted issuers.

AttackWhat the attacker doesRiskDefence
Stolen private key Steal the index, agent, or resolver key → sign legitimately. Crypto can't tell. Index key = whole-system compromise. ⚠ serious Keys in a KMS/HSM (not plaintext), rotation, fast revocation, and 3-key blast-radius isolation. (MVP keeps plaintext keys in data/ — the first thing to fix.)
Compromised / fake index Remap a name → attacker's key + facts. The whole chain then self-verifies, so the client trusts the attacker. Like a rogue Certificate Authority. ⚠ serious did:key alone doesn't save you — the index chose the key. Pin the index key out-of-band, federate (no single index), add transparency logs, and resolve by self-certifying DID so the key is the name.
Key-substitution (skipped cross-check) Supply forged facts and a matching did:key, so they self-verify, if the client trusts the key the document supplies about itself. △ if lazy Cross-check: the facts' did:key must decode to the same key as the index-vouched public_key in the AgentAddr. Never trust a key a document asserts about itself.
Registration abuse — impersonation / Sybil An open /register lets anyone claim any name (impersonate paypal-agent) or flood the index with millions of fake agents. ⚠ serious (MVP) Authentication + rate-limiting + anti-Sybil on registration (§II.G). (MVP /register is open — a real hole.)
Capability spoofing An agent self-advertises capabilities it doesn't have ("certified medical advisor"). A self-signed claim can lie. △ by design Capabilities must be VCs from trusted issuers (audited), not self-asserted. This is the whole point of §II.G — "audited metadata," not "self-advertising."
Malicious / compromised issuer An issuer you trust is compromised → it vouches for malicious agents, and their credentials verify fine. △ governance Issuer revocation, cross-signing between trust zones, reputation — and the verifier chooses which issuers to trust. A governance problem, not a crypto one.
Revoked-credential replay Use a credential that's been killed but hasn't expired yet. △ bounded VC-Status check before calling + short-lived VCs (<5 min). TTL bounds the window; revocation closes it fast.

The split that lands in the room: signing stops forgery; what's left is stolen keys, trusted-but-compromised authorities, and abuse of registration and trust. Those aren't fixed with more signatures — they need key management (KMS), federation, self-certifying DIDs, registration governance, revocation, and trusted-issuer credentials. DNS never had this attack surface because it never tried to prove identity, capability, or trust — only ownership of a name.

Is it affordable?

Can NANDA sustain as an index — or is it too expensive?

The honest worry: NANDA does more than DNS (verification, dynamism, privacy, routing) — so isn't it more expensive to run? The answer is in the architecture: the costly parts are deliberately kept off the index.

  • The core index stays cheap — by design. Records are tiny (120 bytes or less) and rewritten about 10,000× less often than DNS records, so storage and write costs stay low — and reads come straight from edge caches. The index is engineered to be DNS-cheap precisely so it can sustain at scale.
  • Expensive work is pushed to the edges, paid by whoever benefits. Rich metadata (AgentFacts) is hosted and paid for by the agent (self-host / CDN / IPFS); adaptive routing is stateless edge compute; credential verification is offline and cacheable, so it doesn't hit a central server per request. The index never carries that load.
  • Federation spreads the cost. Like DNS root servers run by different operators, the NANDA index is a quilt — enterprises, clouds, and public bodies each run a shard, so no single party funds the whole thing.
  • The added cost vs DNS is real but bounded. A resolution does more crypto than a DNS lookup, but Ed25519 verification is microseconds and happens client-side on cached, signed objects — it scales with clients, not with the index.

The open question (paper §XI.H): who pays for hosting and how — micropayments, staking, or usage tiers — without rent-seeking or gatekeeping. The economics are unsolved, but the technical design ensures the index itself is cheap enough to run as shared public infrastructure, exactly as DNS is.

Verdict

Not a replacement — DNS's wisdom, plus the parts agents need

NANDA isn't trying to kill DNS; it sits beside and above it. It keeps everything that made DNS reliable at planet scale — caching, TTLs, anycast, hierarchy, redundancy, lean records, eventual consistency, signed records — and adds what DNS structurally can't give agents: verifiable capabilities, sub-second revocation, privacy-preserving lookups, and dynamic routing. The leanness that makes DNS cheap is the same leanness that lets NANDA scale affordably. In one line: NANDA is DNS's operating model, re-pointed from static hosts to trust-aware, fast-moving AI agents.
