Visual reference

How agents are discovered, verified, and called.

Six diagrams. The first one traces a real user request — "book me a flight" — from the moment it's spoken to the moment a confirmation comes back, with every signature and every W3C operation labelled. The other five go deeper, and you can refer back to them: how the services fit together, the normal step-by-step lookup, how routing adapts when endpoints move, what happens when someone tampers with the data, and which key signs and verifies what. (All diagrams are drawn in your browser, no server needed.) Below the diagrams, two written sections cover the design rationale (why it's built this way) and the limitations & deferred scope.

Diagram 1 · the headline

End-to-end · "Book me a flight from Mumbai to Singapore on June 10"

A real, concrete user request traced through every layer of the system. The blue band is discovery — finding which agent to talk to. The amber band is the actual call — only reached after verification passes. If any link in the chain breaks, the flight booking never happens.

marks where a W3C / IETF standard governs the operation

🔐 Signature 1 — AgentAddr

  • Signed by: the lean index (K_index, Ed25519)
  • What's signed: the AgentAddr record itself (every field except the signature), converted to a standard byte-for-byte form (JCS) so the signature is reproducible
  • Why it matters: proves the lookup result wasn't substituted en route — the index really did say "here's where to find this agent"
  • Verified by: the client, using the index pubkey fetched at session start

🔐 Signature 2 — AgentFacts VC

  • Signed by: the agent itself (K_agent, Ed25519)
  • Format: W3C Verifiable Credential v2 with DataIntegrityProof, cryptosuite eddsa-jcs-2022
  • Why it matters: proves the agent's capabilities and endpoint URL weren't mutated — even if the facts host or the network is compromised
  • Verified by: the client, using the agent's pubkey carried inside the AgentAddr from Signature 1

The client does all the checking itself — it never just takes a server's word that it's trustworthy. Both signature checks run on the client (the browser uses TweetNaCl, the command-line tool uses PyNaCl) — the same Ed25519 maths over the same canonical (byte-for-byte standardised, RFC 8785) JSON. If anyone tampers with the endpoint URL, the chain breaks and the flight booking call is refused before it leaves the client.

🌐 Standards applied in this flow

W3C VC Data Model v2

Defines the JSON-LD envelope for AgentFacts: @context, type, issuer, credentialSubject, proof.

w3.org/TR/vc-data-model-2.0

W3C Data Integrity

Defines the DataIntegrityProof block: cryptosuite, verificationMethod, proofPurpose, proofValue.

w3.org/TR/vc-data-integrity

W3C eddsa-jcs-2022

The specific cryptosuite that combines Ed25519 signing with JCS canonicalisation. The string eddsa-jcs-2022 appears literally in the proof block.

w3.org/TR/vc-di-eddsa

IETF RFC 8785 — JCS

JSON Canonicalisation Scheme. Guarantees a byte-for-byte identical serialisation across implementations so signatures survive whitespace and key-order differences.

datatracker.ietf.org/doc/html/rfc8785

Every signature in the system uses these four standards. The MVP doesn't ship hand-rolled crypto anywhere.

Diagram 4

Adaptive routing · when endpoints move fast

If the agent's AgentAddr includes an adaptive_resolver_url (the multiregion demo agent does), the client can skip the static endpoint and ask the resolver for a routing token at request time. By default the resolver decides the endpoint (here it just rotates through them in turn; in production it uses GeoIP to pick the nearest one by location); pinning a specific region is an optional override. The token is signed by the resolver and expires after 60 seconds, so even if someone intercepts one, there's only a tiny window in which it could do harm.

Diagram 5

Tamper detection · why mutation fails

A man-in-the-middle can intercept and change the signed credential, but cannot forge a new DataIntegrityProof without the agent's private key. So if the attacker alters the document, the signature no longer matches it — and the client catches the tampering: it re-canonicalizes the changed document, runs Ed25519 verify, and sees the bit-for-bit mismatch.

What each key protects

Index key
Identity of the index itself. If compromised, an attacker could forge AgentAddrs pointing to malicious facts hosts. Mitigation: short TTL + key rotation; index pubkey distribution becomes the trust root.
Agent key
Identity of one specific agent. If compromised, that single agent's facts can be forged. Mitigation: re-register with a new pubkey; old facts become unverifiable instantly.
Resolver key
Identity of an adaptive resolver. If compromised, an attacker could forge routing tokens. Mitigation: 60s TTL means each forgery is worthless almost immediately; resolvers are easy to swap.

Reference

Rotation & TTL — every layer has an expiry

Each signed object carries its own TTL. Nothing trusts forever. This is the central property that makes NANDA safer than DNS, not just different.

Layer Object Default TTL What expires at TTL
Tier 1 · Anchor AgentAddr 3600 s (1 h) Client must re-resolve. New signature with fresh issued_at.
Tier 2 · Metadata AgentFacts VC 300 s (5 m) Client must re-fetch from the facts host. Capabilities may have changed.
Tier 3 · Runtime Routing token 60 s Client must re-dispatch. Endpoint may have moved.

Rationale

Why it's built this way · the decisions behind the diagrams

The whole architecture follows from one simple idea: keep an agent's identity (its name and key, which rarely change) separate from its details (endpoints, capabilities, which change constantly). That way the index can stay tiny and is hardly ever written to, while everything that changes often lives in documents that can be re-published at any moment without touching the index. DNS lumps the two together and takes minutes to hours to update; with trillions of agents that move every few seconds, that simply can't keep up. Keeping them separate means the index has to be rewritten about 10,000× less often, while resolution still completes in under a second worldwide.

Why the index re-signs a fresh AgentAddr on every resolve

The signature covers an issued_at timestamp, so the index stores no signature at rest — it stamps and signs at lookup time. That keeps records cacheable only within their TTL window and means a captured AgentAddr can't be replayed indefinitely. See _row_to_agent_addr in services/index_service/main.py.

Why AgentFacts is a W3C VC but AgentAddr is just a detached signature

AgentFacts is a claim about an agent — capabilities, skills, endpoints — so it gets the full W3C Verifiable Credential envelope (DataIntegrityProof, eddsa-jcs-2022), making it interoperable with any VC-aware verifier. AgentAddr is merely a pointer record the index asserts, so a plain detached Ed25519 signature is sufficient and leaner. Same primitive, two envelopes.

Why the facts host is a deliberately "dumb", untrusted store

The host never verifies signatures on upload — it only checks the format/structure of the document. Trust is entirely the client's job. So even if a facts host (or the network in front of it) is compromised, the attacker still can't forge metadata: the agent's signature won't match the changed document, and the client rejects it. It also decouples hosting from trust, which is what makes the third-party privacy path possible.

Why two facts hosts instead of one

Same service, two roles: an agent-owned primary host (:8001) and a neutral third-party host (:8002) for the privacy path. Running both makes the paper's idea of mixing different registration styles (its "quilt") concrete instead of theoretical, at near-zero extra cost.

Honest scope

Limitations & what's deliberately out of scope

This is an MVP that proves the core flow end-to-end. Several features from the paper are intentionally deferred — each is a real upgrade path, not a dead end. The biggest gap is revocation: the paper makes sub-second VC-Status revocation a centrepiece, and this build relies on TTL expiry alone.

Feature Paper § MVP status Upgrade path
Revocation / VC-Status §VII.D TTL only Add a VC-Status list credential + a status endpoint the client checks during verify.
DID resolution §IV partial did:key is built — the agent's key is recovered from the VC's verificationMethod. Still to add: did:web and DID-based agent_name.
Privacy path transport §VI, §VII.C 2nd HTTP host Real anonymity needs IPFS / Tor / mix-net hosting, not just a separate host.
Enterprise registry quilt §IV.B, Table 1 2 of 6 styles Add enterprise-routed and Web3/DID-marketplace registration types.
Index federation §IV.C Single index Cross-index verifiable links so the "quilt" spans organisations.
Auth on /register — Open Anyone can register an agent; fine for a demo, needs auth + rate-limiting in prod.
Storage §VIII.B SQLite Swap for a sharded KV store to hit the 10k resolves/sec/shard target.

The crypto is real and the Level-1 flow works end-to-end; what's deferred is breadth, not correctness.
