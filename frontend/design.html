Visual reference

How agents are discovered, verified, and called.

Six diagrams. The first one traces a real user request — "book me a flight" — from the moment it's spoken to the moment a confirmation comes back, with every signature and every W3C operation labelled. The other five are technical drill-downs you can refer back to: the system architecture, the standard resolution chain, the adaptive routing path, the tamper-detection failure mode, and the cryptographic trust hierarchy. All rendered client-side as SVG via Mermaid.

Diagram 1 · the headline

End-to-end · "Book me a flight from Mumbai to Singapore on June 10"

A real, concrete user request traced through every layer of the system. The blue band is discovery — finding which agent to talk to. The green band is verification — two independent Ed25519 signature checks that establish the chain of trust before anyone is called. The amber band is the actual call — only reached after verification passes. If any link in the chain breaks, the flight booking never happens.

marks where a W3C / IETF standard governs the operation

🔐 Signature 1 — AgentAddr

  • Signed by: the lean index (K_index, Ed25519)
  • Signed over: JCS-canonical bytes of the AgentAddr minus the signature field
  • Why it matters: proves the lookup result wasn't substituted en route — the index really did say "here's where to find this agent"
  • Verified by: the client, using the index pubkey fetched at session start

🔐 Signature 2 — AgentFacts VC

  • Signed by: the agent itself (K_agent, Ed25519)
  • Format: W3C Verifiable Credential v2 with DataIntegrityProof, cryptosuite eddsa-jcs-2022
  • Why it matters: proves the agent's capabilities and endpoint URL weren't mutated — even if the facts host or the network is compromised
  • Verified by: the client, using the agent's pubkey carried inside the AgentAddr from Signature 1

Neither the index nor the facts host is asked to vouch for itself. Both verifications happen in the client (browser uses TweetNaCl, CLI uses PyNaCl) — same Ed25519 primitive over the same RFC 8785 canonical JSON. If anyone tampers with the endpoint URL, the chain breaks and the flight booking call is refused before it leaves the client.

🌐 Standards applied in this flow

W3C VC Data Model v2

Defines the JSON-LD envelope for AgentFacts: @context, type, issuer, credentialSubject, proof.

w3.org/TR/vc-data-model-2.0

W3C Data Integrity

Defines the DataIntegrityProof block: cryptosuite, verificationMethod, proofPurpose, proofValue.

w3.org/TR/vc-data-integrity

W3C eddsa-jcs-2022

The specific cryptosuite that combines Ed25519 signing with JCS canonicalisation. The string eddsa-jcs-2022 appears literally in the proof block.

w3.org/TR/vc-di-eddsa

IETF RFC 8785 — JCS

JSON Canonicalisation Scheme. Guarantees a byte-for-byte identical serialisation across implementations so signatures survive whitespace and key-order differences.

datatracker.ietf.org/doc/html/rfc8785

Every signature in the system uses these four standards. The MVP doesn't ship hand-rolled crypto anywhere.

Diagram 4

Adaptive routing · when endpoints move fast

If the agent's AgentAddr includes an adaptive_resolver_url (the multiregion demo agent does), the client can skip the static endpoint and ask the resolver for a routing token at request time. The token is signed by the resolver and expires in 60 seconds, so even an intercepted token has a tiny damage window.

Diagram 5

Tamper detection · why mutation fails

A man-in-the-middle can intercept and modify the signed VC, but cannot forge a new DataIntegrityProof without the agent's private key. The client re-canonicalizes the mutated document, runs Ed25519 verify, and sees the bit-for-bit mismatch.

Diagram 6

Trust chain · who signs what, who verifies what

Three independent Ed25519 keypairs guard the system. Compromising any one of them limits the blast radius to that key's scope — the other two stay valid.

What each key protects

Index key
Identity of the index itself. If compromised, an attacker could forge AgentAddrs pointing to malicious facts hosts. Mitigation: short TTL + key rotation; index pubkey distribution becomes the trust root.
Agent key
Identity of one specific agent. If compromised, that single agent's facts can be forged. Mitigation: re-register with a new pubkey; old facts become unverifiable instantly.
Resolver key
Identity of an adaptive resolver. If compromised, an attacker could forge routing tokens. Mitigation: 60s TTL means each forgery is worthless almost immediately; resolvers are easy to swap.

Reference

Rotation & TTL — every layer has an expiry

Each signed object carries its own TTL. Nothing trusts forever. This is the central property that makes NANDA safer than DNS, not just different.

Layer Object Default TTL What expires at TTL
Tier 1 · Anchor AgentAddr 3600 s (1 h) Client must re-resolve. New signature with fresh issued_at.
Tier 2 · Metadata AgentFacts VC 300 s (5 m) Client must re-fetch from the facts host. Capabilities may have changed.
Tier 3 · Runtime Routing token 60 s Client must re-dispatch. Endpoint may have moved.
