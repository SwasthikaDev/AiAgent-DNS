Engineering reference

What each file does, and exactly what runs on a request.

Six parts: a map of the repo, a FastAPI primer (for Django folks), a step-by-step trace of the functions that fire when you resolve and call an agent, a function reference, an honest implemented-vs-paper table, and the CLI commands. The signing/verifying core is one small file (nanda/crypto.py); everything else is plumbing around it.

At a glance

Six services, six ports

Each service is its own FastAPI app on its own port. The ports are set in docker-compose.yml — each service's command: uvicorn … --port N plus its ports: "N:N" mapping — and the browser is pointed at them by frontend/config.js.

PortServiceRole
8000indexLean index — signs AgentAddr; also serves this UI at /ui.
8001facts-primaryAgent-owned facts host (PrimaryFactsURL).
8002facts-privateThird-party facts host (PrivateFactsURL — the privacy path).
8010agent-echoSample agent — echoes input.
8011agent-translateSample agent — mock translator (uppercases).
8020adaptive-resolverAdaptive dispatcher (paper §VI) — hands back short-lived, signed routing tokens that expire after a set time (TTL).

Every service also exposes / for health and /docs for interactive OpenAPI — e.g. http://localhost:8001/docs. It's the same image (Dockerfile) run with a different command: and env per service.

1 · Repository map

What file holds what

Three layers: a shared library (nanda/), the six FastAPI services (services/), and the web UI (frontend/) — plain HTML and JavaScript that runs directly in the browser with no build or bundling step.

Shared library — nanda/

crypto.pyThe heart. Ed25519 + JCS (JSON Canonicalization Scheme — one fixed byte layout so both sides hash the same thing) canonical JSON. canonicalize sign_payload verify_payload (AgentAddr), sign_vc_payload verify_vc (W3C VC), pubkey_to_did_key verify_vc_via_did (did:key).
schemas.pyPydantic wire models: RegisterRequest, AgentAddr, AgentFactsVC, DataIntegrityProof. Defines what travels between services.
cli.pyThe client. Commands: list, resolve, call, verify-did, demo-tamper. Helper _resolve_chain walks index→AgentAddr→AgentFacts and verifies every signature.

Services — services/ (each a FastAPI app)

index_service/main.py :8000The lean index. Routes GET /, POST/PUT /register, GET /resolve/{name}, GET /agents. Signs each AgentAddr via _row_to_agent_addr. Also serves /ui.
index_service/db.pySQLite layer — one agents table. get_by_name insert_agent list_agents upsert_agent.
facts_host/main.py :8001 / :8002A plain storage service for signed credentials — it just saves and serves them and never checks signatures; verification is the client's job. PUT/GET /facts/{id}. One image, two roles (primary + private).
agents/main.py :8010 / :8011Sample agents. POST /echo, POST /translate. The thing you call after verifying.
adaptive_resolver/main.py :8020§VI dispatcher. POST /dispatch → picks an endpoint via _pick_endpoint (geo / capability / round-robin) and returns a signed, TTL-scoped routing token.

Frontend — frontend/ (no build step) & scripts

app.jsBrowser verifier (TweetNaCl). canonicalize verifyEd25519 verifyVC runResolutionCascade + the call & tamper handlers. Mirrors crypto.py byte-for-byte.
index.html / design.html / code.htmlDemo UI · visual diagrams · this page.
config.jsService URLs (overridable via query params).
scripts/bootstrap.pyRegisters the demo agents and publishes their signed VCs. register_agent.
tests/test_crypto.py10 unit tests: detached-sig + VC roundtrip + tamper + did:key.

2 · FastAPI primer

FastAPI, if you already know Django

Every service here is a FastAPI app. Coming from Django, the one big shift is that a route's URL, HTTP method, input validation, and output serialization all live on the function (via a decorator + type hints) — there's no urls.py, no separate serializer class, and the API docs generate themselves.

DjangoFastAPI (in this repo)Note
urls.py + view@app.get("/path")Route + method declared right on the handler. No URLconf.
<path:agent_name>{agent_name:path}Path converter that lets slashes/colons through (our URNs).
Form / DRF Serializer (in)Pydantic BaseModelA typed model as a param → FastAPI parses + validates the JSON body. schemas.py.
DRF Serializer (out)response_model=…Return a model/dict → auto-serialized to JSON in that shape.
request.GET / request.POSTfunction parametersInputs arrive as typed arguments, not off a request object.
raise Http404raise HTTPException(404)Same idea — raise to short-circuit with a status code.
MIDDLEWARE settingapp.add_middleware(…)e.g. CORSMiddleware — same concept as Django middleware.
ORM (models.py)— (raw sqlite3)FastAPI ships no ORM; we use raw SQLite in db.py.
DRF browsable API/docs (auto)Swagger UI generated from the type hints — free, per service.
gunicorn + WSGIuvicorn + ASGIASGI server; FastAPI is async-first. It's the command: line in compose.

The same idea in code — index_service/main.py

app = FastAPI()                          # the ASGI app (≈ a Django project)

# Request body schema — like a DRF serializer / Django form (schemas.py)
class RegisterRequest(BaseModel):
    agent_name: str
    public_key: str
    ttl: int = 3600                        # typed + default, validated for free

# Route + method + output shape, all on the function. No urls.py.
@app.post("/register", response_model=AgentAddr)
def register(req: RegisterRequest):        # body parsed + validated into `req`
    ...
    return signed_addr                       # auto-serialized to JSON as an AgentAddr

# {agent_name:path} ≈ Django's <path:…> — lets the URN's colons through
@app.get("/resolve/{agent_name:path}", response_model=AgentAddr)
def resolve(agent_name: str):
    row = db.get_by_name(agent_name)
    if row is None:
        raise HTTPException(status_code=404, detail="no such agent")
    return _row_to_agent_addr(row)           # build + sign, then return

The mental shift that pays off: in Django you reach into a request object and write out through a serializer; in FastAPI the function signature is the contract — typed inputs come in as arguments, the typed return goes out as JSON, and /docs documents it automatically.

3 · Routing trace

What runs when you call an agent

Tracing Resolve & Call on the multiregion agent with adaptive routing on. The default routing is Auto, so the resolver round-robins — it picks endpoints in rotation. Pinning a region is an override. Each step shows the HTTP hop, the service handler it lands in, and the function chain. The same chain runs in the CLI (_resolve_chain); only the verifier library differs — the browser uses TweetNaCl, the CLI uses PyNaCl.

1
Fetch the index public key.
GET :8000/ → index_service.main root(). The browser caches this key as its trust anchor — the root key every later signature is checked against. app.js fetchIndexPubkeyRich
2
Resolve the name → signed AgentAddr.
GET :8000/resolve/{name} → resolve() → db get_by_name() → _row_to_agent_addr() → crypto sign_payload(). The index signs the record on the fly with its private key.
3
Verify the AgentAddr (in the client).
app.js verifyEd25519(addr, indexPub) → canonicalize() + nacl.sign.detached.verify. Fail → stop. Pass → trust the agent's public_key carried inside the AgentAddr.
4
Fetch the AgentFacts Verifiable Credential (VC) — a signed record of what the agent claims about itself.
GET :8001/facts/{id} → facts_host.main get_facts() (a dumb file read — no verification here).
5
Verify the VC (in the client).
app.js verifyVC(facts, addr.public_key) → strips proof, re-canonicalizes, Ed25519-verifies. (CLI equivalent: crypto verify_vc, or verify_vc_via_did for the did:key path.)
6
Dispatch via the Adaptive Resolver (§VI).
POST :8020/dispatch → adaptive_resolver.main dispatch() → _pick_endpoint() (resolver decides — round-robin by default; geo / capability are overrides) → crypto sign_payload() signs a 60-second routing token.
7
Verify the routing token (in the client).
app.js verifyEd25519(token, token.resolver_pubkey). Fail → refuse to route.
8
Call the verified endpoint.
POST :8011/translate → agents.main translate() → response rendered.

Without --adaptive, steps 6–7 are skipped and the endpoint comes straight from facts.credentialSubject.endpoints.static[0] (step 5). Three signatures total are checked client-side: AgentAddr (index key), AgentFacts VC (agent key), routing token (resolver key).

4 · Function reference

The functions that matter

The ones a reviewer will ask about, grouped by what they do.

FunctionFileDoes
canonicalizecrypto.py / app.jsRFC 8785 JCS — sorted keys, no whitespace, UTF-8. Same bytes on both sides so signatures match.
sign_payload / verify_payloadcrypto.pyAn Ed25519 signature that travels alongside the data (a 'detached' signature, kept separate from the payload). Used for AgentAddr and routing tokens.
sign_vc_payload / verify_vccrypto.pyW3C VC v2 with DataIntegrityProof (eddsa-jcs-2022). Used for AgentFacts. Signs before the proof block exists.
pubkey_to_did_key / verify_vc_via_didcrypto.pydid:key: the public key is packed right into the identifier, so you verify a VC using a key pulled from the credential's own verificationMethod — not a key handed over by the index.
_row_to_agent_addrindex_service/main.pyBuilds + signs an AgentAddr from a DB row at resolve time (fresh issued_at each call).
_pick_endpointadaptive_resolver/main.pyChooses an endpoint by policy: geo region, requested capability, else round-robin.
_resolve_chaincli.pyCLI driver — runs the 5-step verify chain and prints each check.
runResolutionCascadeapp.jsBrowser equivalent — same chain, animated, verified in-page via TweetNaCl.

6 · CLI commands

Driving it from the terminal

The CLI (nanda/cli.py, built with Typer) is the technical surface — it walks the same chain as the web UI and prints every signature check. Run the stack and seed the agents first: 

# one-time setup
docker compose up --build -d        # start the 6-service stack
pip install -r requirements.txt     # deps for the CLI
python scripts/bootstrap.py         # register the 3 demo agents
CommandWhat it does
listList every agent the index knows about.
resolve <name>Walk index → AgentAddr → AgentFacts, verifying every signature. --private uses the third-party host; --show-facts prints the full VC.
call <name>Resolve + verify, then POST a message to the endpoint. -m/--message, --adaptive (§VI resolver), --region.
verify-did <name>Verify AgentFacts by recovering the key from the VC's did:key — no index key copy trusted.
demo-tamper <name>Mutate a signed field and show the client rejecting it.
 
python -m nanda.cli list
python -m nanda.cli resolve urn:agent:demo:echo
python -m nanda.cli resolve urn:agent:demo:translate --private --show-facts
python -m nanda.cli call urn:agent:demo:echo --message "hello"
python -m nanda.cli call urn:agent:demo:multiregion --adaptive --region eu-west
python -m nanda.cli verify-did urn:agent:demo:echo
python -m nanda.cli demo-tamper urn:agent:demo:echo

python -m nanda.cli --help          # all commands
python -m nanda.cli call --help     # flags for one command

Each command is a @app.command()-decorated function in cli.py; the function name is the command, and its typed parameters become the arguments and --flags. Same self-documenting style as the FastAPI services.
