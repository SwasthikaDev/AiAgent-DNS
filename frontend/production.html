3 · The architecture (request path, top to bottom)

  • Global anycast + GeoDNS routes each client to the nearest region.
  • CDN / edge cache serves signed AgentAddr + AgentFacts — absorbs ~95%+ of reads (safe, because they're signed).
  • Regional L7 load balancers spread the remaining ~6 k/s across a stateless index-resolver fleet that autoscales on RPS.
  • Sharded, replicated KV store (consistent hashing on agent_id) — e.g. 32–64 shards, 3 replicas each, multi-region read replicas. 6 k/s origin reads + ~hundreds/s writes is comfortable; far under the 10 k/s/shard target.
  • Facts hosting = object storage + CDN (agent self-publishes); updates never hit the index.
  • Adaptive-resolver fleet — stateless, at the edge / serverless, signs short-TTL routing tokens.
  • VC-Status / revocation service — fast replicated cache + an event stream for sub-second global propagation.
  • Async backbone (Kafka) — registrations, key rotations, revocations, telemetry flow as events; CDC fans them to replicas and triggers edge-cache invalidation.
  • Observability — OpenTelemetry traces + metrics; append-only sampled audit log.

Foundations

Designing for the four "-ilities"

The non-functional requirements that decide whether infrastructure survives contact with reality. Here's how each one shapes a NANDA build.

PropertyWhat it means hereThe levers
Scalability Handle billions→trillions of agents without re-architecting. Horizontal sharding of the lean index (hash by agent_id); stateless services that autoscale; decentralized, index-free facts hosting; read-scale via edge caching + geo-replicas. Lean ≤120 B records keep per-agent cost near-zero.
Maintainability A small team can evolve it safely for years. Small single-concern services (one image, different config); a versioned, forward-compatible schema (JSON-LD @context); a published contract-test suite as the spec; full observability so changes are measurable.
Efficiency Minimum work per resolution; cheap at scale. ~10⁴× fewer writes than DNS (lean records); verify-once-cache-everywhere (signed objects are CDN-cacheable); Ed25519 is small & fast (32 B keys, 64 B sigs); JCS canonicalisation means a signature made once verifies anywhere with no re-signing; TTLs tuned to volatility.
Reliability Stays up and correct under failure. Multi-region redundancy, no single point of failure (federated indexes), resolver fallback rules, idempotent writes, retries/backoff, circuit breakers, SLAs + on-call.

These four trade against each other — e.g. stronger consistency (a reliability/correctness lever) costs latency (efficiency). The next two sections are about managing exactly those trade-offs.

Data layer · storing, moving, transforming scale

Polyglot persistence — the right store for each job

There is no single database. Different parts of NANDA have completely different access patterns, so each gets the store that fits — the classic polyglot persistence approach.

Storing — what lives where

DataStore typeWhy
Lean index (AgentAddr pointers)Distributed key-value store (Cassandra / DynamoDB / etcd)Point lookups by agent_id, huge write throughput, horizontal sharding. No joins or cross-agent transactions → a relational DB is the wrong tool.
AgentFacts (VC documents)Object / document / content-addressed store (S3 / CDN / IPFS)Large-ish signed blobs, read-heavy, agent-self-published, cacheable at the edge.
Revocation statusFast replicated cache (Redis) / VC-Status listMust answer "is this still valid?" in sub-second, globally.
Telemetry & audit logAppend-only log / time-series (Kafka / ClickHouse / immutable ledger)High-volume, write-once, ordered, queried for trust scores and compliance.

Moving — getting data where it's needed

  • Replication across regions (leader-follower or multi-leader) so reads are served locally — the <1 s target is mostly a data-locality problem.
  • Event streaming + Change Data Capture (Kafka): registrations, key rotations, and revocations propagate as events to every edge cache and replica.
  • CRDT-based async updates (the paper's update protocol): conflict-free metadata revision across replicas without coordination — agents publish independently.
  • Anti-entropy / gossip to reconcile replicas; accept eventual consistency bounded by TTL.

Transforming — turning raw data into useful data

  • Canonicalise → sign: every object is JCS-canonicalised (RFC 8785) before an Ed25519 signature — the core transform that makes records verifiable and cache-safe.
  • Row → signed AgentAddr: the index reads a tiny DB row and stamps + signs it on the fly (_row_to_agent_addr) — a read-time transform, so it never stores a stale signature.
  • Telemetry → Trust-Reputation Scores: stream-process raw metrics into reputation over time (§VIII.B, §X).
  • Facts → discovery index: extract searchable metadata from AgentFacts for reputation graphs / directories (§XI.B).

Consistency · the CAP theorem security

Pick your trade-off — and pick it per subsystem

CAP says a distributed system can guarantee at most two of Consistency, Availability, and Partition-tolerance. On a global network, partitions will happen, so P is non-negotiable — the real choice is C vs A when the network splits. NANDA doesn't pick once; each layer picks what fits.

LayerChoiceReasoning
Resolution (index reads)AP — available + eventually consistentBetter to return a signed, slightly-stale-within-TTL AgentAddr than to fail a lookup during a partition. Staleness is bounded (TTL) and detectable (signed), so it's safe. Resolution must never go down globally.
Revocation / securityleans CP — wants strong consistencyYou must not serve a revoked credential. This is the tension: it pulls toward consistency. Reconciled by short TTLs (<5 min VCs) + a fast revocation list — you accept a small, tunable window of inconsistency as the safety/availability knob.
Audit logCP — durable + orderedCompliance and liability need a correct, append-only, ordered record — favour consistency/durability over availability for writes.

The key insight: NANDA can choose AP for resolution because its records are signed (a stale answer can't be a forged answer) and TTL-bounded (staleness is capped). Cryptography is what makes eventual consistency safe here — that's the whole reason the paper signs everything and scopes it with TTLs.

Going deeper (PACELC): even with no partition, there's a latency-vs-consistency trade-off. NANDA chooses Latency for resolution (cache + replicas, eventual consistency) to hit the <1 s target — i.e. it's a PA/EL system: Available under partition, Latency-first otherwise.

Reliability targets · SLI → SLO → SLA ops

Promise reliability in numbers, then defend it

You can't run trusted infrastructure on "it feels fast." Reliability is defined as measured numbers and a budget for failure. Three terms, often muddled:

  • SLI (Indicator) — a measured metric. e.g. "% of resolves that return a valid signed AgentAddr in <1 s."
  • SLO (Objective) — the internal target for an SLI. e.g. "99.95% of resolves succeed." This is what the team designs and alerts against.
  • SLA (Agreement) — the external, contractual promise to customers, with consequences (credits) if breached. Deliberately set looser than the SLO, so you have a buffer before you owe anyone.

Error budget = 100% − SLO. A 99.95% SLO means a ~22-minutes-a-month budget of allowed failure. You spend it on releases and risk; when it's exhausted, you freeze features and fix reliability. It's the mechanism that turns "move fast" vs "stay up" from an argument into a number.

What to measure (SLI)Example SLOTied to paper target
Resolution success rate99.95%Discovery must work — the index is the anchor.
Resolution latency (p99)< 1 sThe paper's <1 s global-resolution target.
Revocation propagation< 1 s globalSub-second revocation (§VII.D).
Index throughput10k updates/s/shard§II.D scalability target.
Signature-verify success≈ 100%A failed verify is a security event, not noise.

SLOs drive the alerts and the error budget; the SLA is the subset you're willing to put in a contract. Measure SLIs with the OpenTelemetry pipeline from section 6.

Backup, failure & disaster recovery ops

Assume every component fails — design for it

Two questions drive disaster recovery: RPO (Recovery Point Objective — how much data can we afford to lose?) and RTO (Recovery Time Objective — how fast must we be back?). They're different per layer.

Backups — per store

DataBackup strategyRPO / RTO
Lean index (KV)Cross-region replicas + periodic snapshots + point-in-time recoveryRPO≈0, RTO low — it's the anchor; losing it stops all resolution.
Signing keysKMS/HSM with managed durability + recovery; publish key historyNever lose; never expose. Old objects must still verify after rotation.
AgentFactsObject versioning / content-addressed (IPFS); agents can re-publishSelf-healing — the agent owns the source of truth, so loss is recoverable.
Audit logAppend-only, immutable, replicatedRPO≈0 — compliance can't tolerate gaps.

Failure modes & response

  • Node failure → replicas take over (health checks + load balancer); no data loss.
  • Region failure → multi-region failover + anycast routes clients to a healthy region.
  • Data corruption / tampering → signed records make it self-detecting (verify fails); restore from backup.
  • Key compromise → rotate + revoke; blast radius is isolated by the three independent keypairs (index / agent / resolver), so one compromise doesn't sink the system.
  • Adaptive resolver offline → fallback rules (static endpoint, retry in another region) — the call still completes (§VI.D).
  • Whole-index outage → because AgentAddrs are signed and cacheable, clients keep resolving from their caches within TTL. The system degrades gracefully instead of going dark.

Make DR real

  • Test restores regularly — an untested backup is a hope, not a backup.
  • Chaos engineering / game days: kill a region in staging and watch failover + the error budget.
  • Runbooks for each failure mode so on-call isn't improvising at 3 a.m.

The architectural win: signed + TTL-bounded + cacheable + decentralized-facts means NANDA is inherently resilient — much of the demo's design (sign everything, cache freely, agents own their facts) is exactly what makes graceful degradation possible. The crypto isn't just security; it's availability.

1 · Scale & performance scale

Keep the index lean; scale reads at the edge

  • Sharded, replicated KV store instead of SQLite. Partition by a hash of agent_id (consistent hashing); the index is a key→tiny-record lookup, which shards trivially. Target the paper's 10k updates/sec/shard (§II.D).
  • Stateless services that scale horizontally. Index resolvers, facts hosts, and adaptive resolvers hold no per-request state, so they sit behind a load balancer and autoscale on request volume (§VIII.B "load-adaptive resolvers").
  • Keep records ≤120 B (§II.A). Store credential hashes/pointers, not full keys or metadata — that's what gives ~10⁴× fewer writes than DNS and keeps shards cheap.
  • Anycast + geo-distributed read replicas so a lookup is answered near the client (the <1 s target is a latency problem, not just throughput).
  • Separate the read and write paths. Resolution is overwhelmingly read; registration/rotation is the rare write. Optimise them independently.

Performance · throughput vs latency scale

Two different numbers, optimised two different ways

Easy to confuse, but orthogonal: latency is how long one request takes; throughput is how many requests you sustain per second. A system can have huge throughput and terrible tail latency, or the reverse — you design for each differently.

MetricMeansNANDA targetHow you get it
Latency time for one resolution < 1 s global (p99) Edge caching + anycast/geo-replicas (answer near the client); client-side verification (no server round-trip); signed AgentAddr is cacheable, so the index isn't on the hot path; minimise hops.
Throughput ops/sec sustained 10k updates/s/shard;
reads ~unbounded via cache		 Horizontal sharding (adds write capacity linearly); stateless autoscaling; reads served from cache (never hit origin); async I/O + connection pooling.

Optimise the tail, not the average. A p50 of 40 ms is meaningless if p99 is 4 s — one cold cache or far-region hop blows the <1 s budget, and the tail is what users actually feel. SLOs are set on p99 for this reason.

Where the <1 s budget goes (one resolution)

  • Index lookup → ms-scale, usually from an edge cache or nearby replica.
  • Fetch AgentFacts → ms-scale, CDN-cached signed VC.
  • Verify signatures → microseconds, Ed25519, client-side — effectively free and doesn't load any server.
  • Optional adaptive dispatch → ms-scale, stateless edge resolver.

They're coupled under load (Little's Law: concurrency = throughput × latency). If latency spikes, in-flight requests pile up and throughput collapses. That's why caching is the highest-leverage lever — it cuts both: a cache hit is low-latency and removes load from the origin, raising throughput at the same time.

2 · Caching & CDN scale

The highest-leverage lever — cache aggressively at the edge

Caching cuts latency and offloads the origin (raising throughput) at the same time — no other lever does both. DNS proves the model (it's essentially one giant distributed cache), and NANDA inherits it with a twist: because every object is signed, it's safe to cache even on edges you don't fully trust.

The cache layers — closest to the user wins

LayerCachesLifetime
Client (agent / browser)AgentAddr it already resolvedAgentAddr TTL (≈1 h)
CDN / edge PoPsigned AgentAddr + AgentFacts VC, near the userper-object TTL
Regional replicathe index shard's data, in-regionreplicated
Origin (index shard / facts host)source of truthauthoritative

A request is answered by the closest layer with a fresh copy; most never reach the origin — which is exactly how DNS serves billions of lookups cheaply.

What a CDN buys you

  • Latency — content served from a PoP geographically near the user (the <1 s lever).
  • Throughput — origin offload: a 95% cache-hit ratio means the origin sees ~1/20th the traffic.
  • Availability — the edge keeps serving cached (signed, within-TTL) objects even if the origin is down → graceful degradation.
  • DDoS absorption — the CDN's capacity soaks up floods before they reach your origin (the lesson from the 2016 Dyn outage).

Why this is safe — the NANDA twist

A CDN edge is infrastructure you don't fully control. Normally you'd worry it might tamper with cached content. But every AgentAddr and AgentFacts VC is Ed25519-signed and verified client-side, so a malicious or buggy edge cannot forge a response. That's what lets NANDA push signed objects to any edge — even third-party CDNs or IPFS — with no trust problem. (Routing tokens have a 60 s TTL, usually too short to bother caching.) Signed objects can even be redistributed peer-to-peer by clients within their TTL — the paper's "2N handshake," where the index is touched once.

Cache invalidation — the genuinely hard part

  • TTL expiry — the default; automatically bounds staleness. Tune per volatility (healthcare vs demo, §V.E).
  • Revocation list — the "kill it now" override when TTL is too slow; a VC-Status list gives sub-second invalidation (§VII.D).
  • Explicit purge — push a purge to the CDN for an urgent change; but lean on short TTLs so you rarely need it.

Operate it well

  • Cache-hit ratio is a top SLI — it drives both latency and origin cost directly.
  • Guard against cache stampede (thundering herd): when a hot key expires, many clients miss at once and hammer the origin. Mitigate with request coalescing / stale-while-revalidate.

TTL · the caching control knob, in detail scale

TTL is the single dial that trades freshness against cost

TTL (Time-To-Live) = how long a cached copy is treated as fresh before it must be re-fetched or re-validated. It's the most important number in the whole caching story, so it's worth understanding in full.

How it works

When any layer caches a signed object, it starts a timer set to that object's TTL. Within the TTL, every request is answered from cache — fast, no origin hit. After the TTL, the entry is stale and the next request triggers a re-fetch (or a revalidation). The TTL is carried inside the signed object itself, so it can't be tampered with — the paper enforces that records are "resolved and cached only within their TTL window" (§VII.E).

The three NANDA tiers — and why each value

LayerTTLWhy that length
AgentAddr (index)1–6 hIdentity + pointers are stable → cache long, spare the index.
AgentFacts (metadata)5–15 mCapabilities/endpoints change more often → medium freshness.
Routing token (endpoint)30–60 sEndpoints move every few seconds → very short, near-real-time.

The core trade-off

  • Long TTL → fewer lookups, lower origin load, lower latency on a hit — but a longer staleness window: a change (or a revoked credential) can linger up to one TTL.
  • Short TTL → fresher data, faster propagation — but more re-resolution, more origin traffic, higher cost.
  • The rule of thumb: worst-case staleness ≈ TTL, and propagation time ≈ TTL. You're literally choosing how stale you tolerate.

The details that matter in practice

  • TTL is your consistency knob (CAP tie-in). Shorter TTL = closer to strong consistency, at higher cost; longer TTL = more eventual. It's how you dial the AP/CP balance per object.
  • Layered TTLs compound. Each tier has its own TTL; effective freshness is the shortest relevant one. Advice from the paper: set cache expiry shorter than the weakest credential's TTL (§VII.D).
  • Revocation beats TTL when you can't wait. TTL alone can't do sub-second — for "kill it now," a VC-Status list invalidates ahead of expiry. TTL bounds the default staleness; revocation handles emergencies.
  • Tune by criticality (§V.E). A healthcare or financial agent gets short TTLs (fresher, safer); a demo agent gets long TTLs (cheaper). Per-agent, per-trust-zone.
  • Negative / error caching. Cache "not found" and failures with a short TTL too (like DNS caches NXDOMAIN) so a missing name doesn't hammer the origin on every retry.
  • stale-while-revalidate. Serve the slightly-stale copy immediately while fetching a fresh one in the background — hides the re-fetch latency at expiry and smooths load.
  • TTL jitter. Add randomness so many cached entries don't expire at the same instant — prevents a synchronized cache stampede hammering the origin.
  • Open question (§XI.E): who sets the TTL — the agent, the registry, the credential authority, or a policy engine? And can TTLs be ephemeral / task-conditioned (tied to a session or mission)? Unresolved in the paper.

Hierarchical delegation · the DNS model, applied scale

How DNS scales by delegation — and how NANDA borrows it

DNS serves ~10 B lookups/day for a reason: no single server knows everything. Authority is delegated down a hierarchy, each level independently operated, replicated, and cached. NANDA's "quilt" is the same idea — and it's the key to distributing load, governance, and trust at agent scale.

The DNS hierarchy (how a name resolves)

  • Root servers (.) — know only where the TLD servers are. Tiny dataset, 13 "letters" → thousands of anycast instances, cached everywhere.
  • TLD servers (.com, .org, .in) — know which authoritative server owns each SLD under them.
  • Authoritative / SLD (example.com) — actually holds the records (A, MX, …).
  • Recursive resolver (your ISP / 8.8.8.8) — walks root → TLD → authoritative once, then caches so it rarely has to again.

Why this scales: each level is small and cacheable; load and authority are partitioned by name; the heavily-hit root holds almost nothing. Adding a new domain never touches the root.

So a resolution can be multi-hop: AgentName → NANDA index → enterprise registry → AgentFacts (the paper's "Enterprise Registry Resolution" path, §IV.A). The index need not know every agent — it delegates, exactly like the DNS root.

Why delegation is a scaling superpower

  • Distributes write load — each org registers/updates its own agents in its own registry; no central write bottleneck (ties to the ~100 k/s of updates staying off the root index).
  • Distributes governance & liability — each zone sets its own policy, credential schemas, and revocation (§VII.B trust zones), with cross-signing between them.
  • Independent scaling & failure isolation — a busy or broken enterprise registry doesn't take down the root or other zones.
  • Keeps the root lean & cacheable — like DNS, the most-hit tier holds the least, so it caches beautifully and rarely changes.

Proxy servers · forward & reverse scale security

Two kinds of middleman — and NANDA uses both

A proxy is a server that sits between two parties and forwards traffic on their behalf. Which side it hides determines its type — and the NANDA architecture leans on both, one for privacy and one for routing.

TypeSits in front ofHidesIts NANDA role
Forward proxy the client the client's identity The privacy path: a neutral relay / Tor / mix-net node fetches the PrivateFactsURL so the agent's host never sees the requester's IP — "requester anonymity" (§VII.C).
Reverse proxy the servers the backend topology The adaptive resolver (the paper names Envoy / NGINX, §VI.E), the API gateway in front of every service, and the CDN edge — all reverse proxies.

Forward proxy → the privacy path

When a client resolves via PrivateFactsURL, it fetches the agent's metadata through a neutral forward proxy. The agent's domain is never contacted, and only the proxy sees the request — so the requester stays anonymous and access patterns aren't leaked to the agent or a passive observer. Chaining relays (Tor / mix-net) further defeats traffic-analysis correlation. This is exactly the "requester anonymity + access decoupling" of §VII.C.

Reverse proxy → fronting the services

Every service should sit behind a reverse proxy / API gateway that handles TLS termination, load balancing across replicas, rate-limiting (the /register protection), request routing, and DDoS shielding — the paper points to Envoy/NGINX for the adaptive resolver, and a CDN is just a globally-distributed reverse-proxy cache (the caching section above). It's also the natural place to enforce mTLS between services, centralise auth, and emit per-request telemetry, since every request flows through it — one clean control point.

3 · Trust, keys & security security

Real key management and revocation

  • Private keys in a KMS/HSM, never plaintext files. The MVP stores keys in data/*.json — in production those move to a managed secrets store with access control and rotation.
  • Sub-second key rotation & revocation (Abstract, §VII.D): short-lived VCs (<5 min) + a VC-Status revocation list clients check during verification. Rotate the index/resolver signing keys on a schedule; publish key history so cached objects still verify.
  • DID-based identity (§IV): resolve the agent's key from its did:key/did:web (this MVP does did:key), so the index stores a hash, not a copy of the key — rotation doesn't require an index write.
  • Authn + rate-limiting on writes. /register must be authenticated and throttled to stop spam/abuse and Sybil registration (§II.G).
  • Federated trust zones & cross-signing (§VII.B): credential authorities govern their own zones; clients set trust policy (whitelisted issuers, threshold verification, Trust-Reputation Scores). No hardcoded trust anchors.
  • Defence in depth: mTLS between services, signed everything, append-only audit logs for tamper-evidence (§II governance).

4 · Privacy & split-horizon security

Hide who is asking, not just what is signed

  • Real privacy-path hosting (§VII.C): the MVP's "private" facts host is just a second HTTP service. In production it's IPFS / a neutral gateway / Tor-style relays so the agent's domain is never contacted and the requester stays anonymous.
  • Dual-path resolution via PrimaryFactsURL + PrivateFactsURL, with cacheable or mix-net-broadcast requests to defeat traffic-analysis correlation (§E).
  • Split-horizon: return different answers depending on who/where is asking, for enterprise governance — without leaking the query.
  • Data minimisation / least-disclosure; longer term, ZKP-based capability proofs (§X) so an agent proves compliance without revealing full certificate chains.

5 · Reliability & resilience ops

Graceful degradation, no single point of failure

  • Multi-region, redundant deployment with health checks and automatic failover. The index is a globally distributed federated system (§IV.C), not one box.
  • Fallback rules for the adaptive resolver (§VI.D): if the resolver is unreachable, fall back to a static endpoint, retry with backoff in another region, or escalate to credential-based access — never hard-fail the call.
  • Idempotent writes, retries with backoff, circuit breakers between services so one slow dependency doesn't cascade.
  • SLAs + on-call. Once external builders depend on a public testnet, reliability is the product — define SLOs (resolve success rate, p95 latency) and a real rotation.
  • DDoS resilience: rotating/shuffle-sharded endpoints and signed routing tokens so infrastructure can be obfuscated and shuffled under attack (§VI).

6 · Observability & audit ops

You can't operate what you can't see

  • OpenTelemetry across every service (§VIII.B). A single resolution spans index → facts host → resolver → agent; distributed tracing ties those hops into one trace so you can find where the <1 s budget is spent.
  • The metrics that matter: resolves/sec, p50/p95/p99 latency, error rate, cache-hit ratio, revocation propagation time, signature-verify failures.
  • Append-only, tamper-evident audit logs for governance and liability — proving compliance and assigning blame when an agent misbehaves across borders (§II).
  • Behaviour-based trust scoring fed by telemetry (§VIII.B, §X) to compute reputation over time.

7 · Hosting & storage scale

Decentralised, index-free metadata

  • AgentFacts on object storage / CDN / IPFS, not local files. Agents sign and self-publish their VCs to self-selected hosts and just point the lean index at the URL — no index write to update metadata (§II.D).
  • Flexible hosting model (§XI.A): self-hosting (provenance, but reveals access patterns), third-party hosting (privacy, but host-trust dependency), or neutral public infra. Support all three; let publishers choose by trust/latency/control.
  • CRDT-based update protocol (Abstract) for conflict-free, asynchronous metadata revision across replicas.

8 · Federation & governance security

A quilt of registries, not one authority

  • Cross-index federation (§IV.C): independent indexes operated by enterprises, clouds, or public bodies, linked with verifiable credentials and registry-to-registry bridges. "Quilt" only becomes real with federation.
  • Diverse registration models (§IV.B, Table 1): NANDA-native, enterprise-routed, and Web3/DID-marketplace agents — with consistent metadata standards across boundaries.
  • Standards alignment (§X): drive the AgentFacts schema through IETF/W3C, keep the trust primitive (signed VCs over canonical JSON) portable so it interoperates with A2A, MCP, NLWeb, ANS/ANP.
  • Governance model: who decides what goes in the spec, revocation/dispute protocols, multi-jurisdictional policy — sketched as a steering committee with rotating representation.

9 · Delivery & engineering practice ops

How the team ships it safely

  • Kubernetes (not docker-compose) for real orchestration: autoscaling, rolling/blue-green deployments (the endpoint-agility the paper assumes, §II.C), self-healing.
  • Infrastructure as code + CI/CD: every service tested, linted, and deployed through a pipeline; secrets via the KMS, never in the repo.
  • A published contract-test suite any compliant index / facts host / client must pass — the reference implementation is the spec, so interop is enforced by tests, not prose.
  • Independent security review of the reference implementation before anyone depends on it; threat-model the trust and revocation paths.
  • Versioned, forward-compatible schema: JSON-LD @context lets AgentFacts evolve without breaking old clients (§I).
